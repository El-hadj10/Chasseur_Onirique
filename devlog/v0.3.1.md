## What this version adds

Two followups from the v0.3.0 launch, both validated by the
`thinker-with-files-gemini` design review:

1. **The fixer** ŌĆö a post-processing step in the pentest agent that
   generates proposed patches for findings. Gated by
   `CHASSEUR_ONIRIQUE_PENTEST_FIX=1` (off by default). The output is
   `docs/pentest/<ISO>-fixes.md` ŌĆö a markdown file with embedded
   ````diff` code blocks for auto-fixable items and "Manual action
   required" notes for everything else. The user reviews and applies
   manually (or with `git apply` on the diff blocks).
2. **HeuristicPlanner security routing** ŌĆö prompts containing
   `security` / `audit` / `pentest` / `vulnerability` / `secret` /
   `CVE` now route step 2's `suggestedAgents` to `['pentest',
   'thinker']` instead of just `['thinker']`. This makes the pentest
   discoverable from the main CLI without going through
   `npm run pentest`.

## The fixer design

The design question was: should the fixer be a 5th parallel sub-check
(like SAST/deps/secrets/network) or a post-processing step?

The `thinker-with-files-gemini` review pushed back on the 5th-sub-check
design: a parallel sub-check would race against the other 4 (which are
still emitting findings while the fixer tries to read them). The fix is
to run the fixer AFTER `Promise.allSettled` resolves, so the `findings`
array is fully populated. That's what the code does.

### Auto-fixable scope

The regex-based SAST scanner cannot reason about the surrounding code,
so any auto-fix that touches control flow is dangerous (e.g., removing
an `eval()` call leaves a dangling assignment; replacing `createHash('md5')`
with `createHash('sha256')` breaks the rest of the code if the hash
length is consumed elsewhere). The fixer is therefore restricted to
deterministic, line-level changes:

- **SAST `console.*`** ŌĆö remove the line. The rest of the file is
  untouched.
- **SAST `TODO` / `FIXME` / `HACK` markers** ŌĆö remove the comment.
  Same: rest of the file is untouched.
- **Deps deprecated package** ŌĆö remove the JSON key from
  `package.json`. The diff is targeted (3 lines of context before and
  after), so it's valid for `git apply`.

Everything else gets a "Manual action required" note with a one-line
explanation of what the human needs to do:

- **SAST `eval`, `new Function`, `child_process.exec`, `md5`/`sha1`,
  hardcoded IPs** ŌĆö the replacement depends on the surrounding code.
  The fixer points the user to the file:line and explains the
  trade-off.
- **Deps suspicious range (`*`, `latest`)** ŌĆö the version is
  unknowable. The fixer tells the user to run
  `npm view <pkg> versions` and pick a current version.
- **Secrets** ŌĆö rotation requires a human + `git-filter-repo` (or
  equivalent). The fixer explains this.
- **Network** ŌĆö firewall review. The fixer points the user to the
  port and the `localhost:<port>` evidence.

### The "re-read the file" requirement

The `Finding.evidence` field is pre-truncated to 200 chars. A diff
generated from a truncated line would be invalid (the surrounding
context is missing). The fixer therefore re-reads the original file
from disk using `Finding.file` and `Finding.line`, extracts 3 lines of
context before and after the target, and builds a proper unified diff
hunk with the correct `@@ -start,count +start,count @@` header.

This is the "Big Missing Piece" the thinker called out. Without it,
`git apply` would reject the patch.

### File naming and location

The output file is `docs/pentest/<ISO>-fixes.md` (NOT `.patch`),
because the content is markdown with embedded diff blocks. A `.patch`
extension would mislead `git apply` into trying to parse the markdown
wrapping as a diff header, which it isn't.

The file goes inside `docs/pentest/` (the same directory as the
NDJSON and per-run MD), so all pentest artifacts are co-located.
`docs/pentest/` is now in `.gitignore` so the fixes file is not
accidentally committed.

### Exit code

The CLI still exits 1 if there are critical/high findings, even when
`--fix` mode is enabled. The reasoning: a fix that exists on disk is
not the same as a fix that has been reviewed and applied. The CI gate
remains "code is insecure until a human reviews the report."

## The HeuristicPlanner security routing

The `HeuristicPlanner` had 3 regex patterns (investigate, refactor,
research). I added a 4th: `security` / `audit` / `pentest` /
`vulnerability` / `secret` / `CVE`. When matched, step 2's
`suggestedAgents` becomes `['pentest', 'thinker']` instead of just
`['thinker']`.

This is a small change (~10 lines), but it makes the pentest agent
discoverable from the main CLI. Before this, a user had to know that
`npm run pentest` existed; now, a prompt like "audit the auth module
for security issues" automatically includes the pentest agent in the
plan.

The test for this is in `planner.test.ts` (2 new tests: one for
`security`, one for `CVE`).

## Safety model (unchanged from v0.3.0)

- The fixer proposes but never applies. The output is a markdown file
  the user reviews.
- All secret redaction still happens inside the secrets sub-check,
  BEFORE the fixer runs. The fixer never sees a plaintext secret.
- The fixer is OFF by default. The env var is opt-in.

## What's next (v0.4 ideas)

- The fixer could learn to handle more SAST patterns (e.g., replace
  `var` with `const`/`let` ŌĆö but this is risky without an AST parser).
- The HeuristicPlanner could route `performance` / `benchmark` /
  `profiling` prompts to a new `profiler` agent (does not exist yet).
- The pentest agent could be wired into the `code_reviewer` step so
  the reviewer automatically runs pentest before approving a change.
- A `--baseline <file>` flag for the pentest CLI to diff findings
  against a previous run, so the CI gate is "no NEW findings" not
  "no findings at all."
