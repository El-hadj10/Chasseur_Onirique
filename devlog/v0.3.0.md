# Devlog v0.3.0 — the pentest agent

> (c) 2026 El-hadj Ousmane — MIT License (see ../LICENSE)

## What this version adds

A `pentest` agent ships as the 8th specialized agent in Chasseur
Onirique. It runs four sub-checks in parallel inside one agent and
streams findings to `docs/pentest/<ISO>.ndjson` (append-only, one JSON
line per finding) plus a final `docs/pentest/<ISO>.md` summary. The
"au fur et à mesure" feel comes from a synchronous `reportFinding`
callback that each sub-check calls the moment it spots something.

A standalone CLI (`npm run pentest`) mirrors the `self-check` pattern:
a custom one-step plan with just the pentest agent, a top-level
`docs/pentest/summary.md`, and exit 0 (clean) or 1 (critical/high
finding).

## The four sub-checks

| Check      | What it does                                              | Reads |
| :--------- | :-------------------------------------------------------- | :---- |
| `sast`     | Pattern-based code scan (eval, shell, weak crypto, etc.)  | source files |
| `deps`     | `package.json` audit (deprecated, suspicious ranges)      | `package.json` |
| `secrets`  | Regex scan with mandatory pre-redaction                   | text files |
| `network`  | Localhost listening ports via `netstat -an`               | nothing (gated) |

Three of the four are pure read-only scans of the filesystem. The
fourth (`network`) spawns a single process (`netstat -an`) and reads
its output. Nothing in the pentest pipeline ever mutates project
state.

## Why one agent, not four agents

The first instinct was to add 4 agents to the registry and have the
planner fan them out. The `thinker-with-files-gemini` review pushed
back, and the design held up under the criticism:

- **Encapsulation.** The 4 sub-checks share one NDJSON file and one
  per-run MD summary. That coordination belongs inside one agent —
  the agent IS the file-streaming coordinator.
- **AgentName pool.** Adding 4 hyper-specific security scouts
  (`sast_scout`, `deps_auditor`, `secret_scanner`, `port_scanner`)
  would dilute the union with names no one will ever plan against.
  The planner still only ever picks `pentest` as a unit.
- **CLI ergonomics.** `npm run pentest` is one command. The
  alternative — `npm run pentest -- --agents sast,deps,secrets,network`
  — is a UX regression for a feature the user wants to be one button.

## The streamed-reporting trick

The orchestrator's `runStep` does `Promise.all` over the step's
`suggestedAgents` and expects a final `AgentOutput` from each. There
is no way to push findings out of an agent mid-flight. So the
"au fur et à mesure" feel is achieved by **pushing findings down to
disk from inside the agent**, not up to the orchestrator:

```ts
const reportFinding = (f) => {
  findings.push({ ...f, id, detectedAt });
  if (live) appendFileSync(ndjsonPath, JSON.stringify(finding) + '\n');
  log.warn(formatFindingLine(finding));
};
```

Each sub-check calls `reportFinding` synchronously the moment it
discovers something. The user sees findings scroll past in real time
in the terminal AND in the NDJSON file, but the orchestrator still
sees a single final `AgentOutput` at the end. The MD summary is
rewritten at the end (NOT appended during) to avoid the
write-write race between the 4 parallel sub-checks.

The `Promise.allSettled` (not `Promise.all`) is intentional: one
sub-check rejecting does not kill the others. The orchestrator still
gets a report with the surviving findings plus the error list.

## The 3 corrections from the design review

The `thinker-with-files-gemini` review caught three things I would
have shipped wrong:

1. **File layout.** My first draft put the sub-checks in
   `src/pentest/checks/`. The reviewer pointed out that the project's
   meta-skill says "agents live in `src/agents/`." Sub-checks are not
   agents, but they belong alongside the agent that owns them. Moved
   to `src/agents/pentest/`.

2. **Mandatory secret redaction.** I had planned to redact secrets
   inside the agent, after the sub-check reported them. The reviewer
   called this out as a leak vector: if the sub-check throws an
   exception mid-report, the unredacted match is still in memory. The
   fix is to redact **inside the sub-check**, the moment a match is
   found. The agent's `reportFinding` callback never sees a plaintext
   secret. This is the single most important safety property in the
   whole agent.

3. **Cross-platform `netstat` instead of `ss`/`lsof`.** I had
   planned to use `ss -tlnp` (faster, more modern) and fall back to
   `lsof -i`. The reviewer noted that `ss` is iproute2-only and not
   on macOS by default; `lsof` is not always installed on minimal
   containers. `netstat -an` is on all three. Bonus: it has a
   consistent two-dialect output across macOS/BSD and Linux that
   can be parsed with a single regex.

## Safety model

- **Read-only by construction.** No shell exec that mutates state.
  The only process spawn is `netstat -an`.
- **Env-var-gated network probe.** Mirrors the
  `CHASSEUR_ONIRIQUE_LIVE=1` pattern. Off by default; set
  `CHASSEUR_ONIRIQUE_PENTEST_NET=1` to opt in. When off, the
  network check still reports an `info`-level "skipped" finding so
  the report shows the gate state.
- **Mandatory pre-redaction.** Secrets are masked inside the
  sub-check. The agent's own log + the NDJSON + the MD all carry
  masked values only.
- **Reporting tool, not a mutation tool.** The agent does NOT respect
  `ctx.dryRun`. The orchestrator's `dryRun` flag is for the agents
  that mutate project state (basher). The pentest agent's writes are
  report persistence (NDJSON + MD), not project mutation — so it
  always writes its per-run files, unless
  `CHASSEUR_ONIRIQUE_PENTEST_WRITE=0` is set. This is the design
  choice that makes the "au fur et à mesure" stream visible by
  default; without it, a `npm run pentest` invocation through the
  orchestrator (which defaults `dryRun` to `true`) would silently
  produce no files.
- **No new dependencies.** The whole agent + 4 sub-checks is
  ~500 lines of zero-dep TypeScript. The 4 sub-checks use Node's
  stdlib (`fs`, `child_process.spawnSync`) only.

## What I deliberately did NOT do

- **No `--fix` flag.** The pentest agent reports findings; it does
  not apply remediations. A fix-it mode would re-introduce the
  shell-exec surface that the basher deny-list exists to contain.
  Maybe v0.4.
- **No network call to a vulnerability database.** The deps check
  uses a hand-curated list of deprecated packages, not the npm
  registry or the GitHub Advisory Database. That would be a
  network round-trip per check — and the agent is supposed to be
  hermetic. Maybe v0.4 with an opt-in `--enriched` flag.
- **No custom rulesets.** The patterns are baked into the sub-checks.
  A `pentest.config.json` would be nice, but it would also be a
  long way to fall if a user's overrides silently neuter the SAST
  check. Maybe v0.4 with a documented "you break it you buy it"
  warning.

## What's next (v0.4 ideas)

- `pentest --fix` mode (env-gated, dry-run by default, with a
  per-fix review step in the CLI).
- A vulnerability database callout in the `deps` check (opt-in,
  hermetic by default).
- A custom ruleset file (`.pentestrc.json`) with a documented
  "you break it you buy it" warning.
- Integration with `code_reviewer` so the orchestrator can chain
  `pentest` → `code_reviewer` automatically on greenfield code.
- A `--baseline <file>` flag to diff findings against a previously
  captured report, so the CI gate is "no NEW findings" not "no
  findings at all."

The pentest agent is feature-complete for v0.3.0. The shell of the
feature is in; the depth is the next version.
