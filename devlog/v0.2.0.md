# Devlog: Chasseur Onirique v0.2.0

*How a 1 500-line multi-agent orchestrator earns the right to call itself one.*

I started Chasseur Onirique on a Tuesday and shipped the first commit on a
Friday. Four days, one rebrand, and a few hard calls about what a
"multi-agent orchestrator" actually has to do. This devlog is the
postmortem of those calls — the trade-offs I made, the debts I took on,
and the ones I refuse to take on even if the next person to read this code
asks nicely.

## What the project is, plainly

Chasseur Onirique is a small Node.js CLI that takes a prompt in, fans it
out to seven specialized agents, and writes a structured report out. The
seven agents are: `file_picker`, `code_searcher`, `basher`, `researcher_web`,
`researcher_docs`, `thinker`, `code_reviewer`. The "fan-out" is real but
bounded — three parallel agents per step, four steps per plan, all
heuristic. There is no LLM in the loop.

That last sentence is the most important design choice in the project, and
it deserves its own section.

## The 4 laws

There are four invariants that the codebase enforces and the docs
document. They're not aspirational — they're load-bearing. Violate one and
the system stops being Chasseur Onirique.

**Law 1: a plan is committed before any action.** The `Orchestrator.run()`
method takes a `Plan` as its second argument. The plan is built by a
`PlannerStrategy` (the only interface the orchestrator depends on for
decision-making), and once built, the orchestrator walks it without
re-deciding. The `HeuristicPlanner` is the dumb default: regex-dispatch
on the prompt, 3 patterns yielding 4 different 4-step plans. Replacing it
with an LLM is a one-line change to the registry. Not doing so is also a
choice.

**Law 2: sub-agent outputs become parent context through one seam.** That
seam is `runStep()` in `src/orchestrator.ts`. Every agent's `notes` map is
merged into the parent's `Context`; every agent's `references` list is
appended. Nowhere else writes to the context. This means the entire
"memory" of the system is a 50-message rolling window plus a typed
`Map<string, unknown>` of scoped notes. No external database. No
file-based persistence. Nothing.

**Law 3: the human is the final gate.** `code_reviewer` is the last agent
in every plan. It runs after a change is applied. It returns a verdict
(approve / request-changes) and a list of issues. The orchestrator logs
the verdict but does not block on it. The user — the human sitting at
the terminal — decides what to do with it. This is the part where most
"AI agent" demos quietly fail. Chasseur Onirique does not auto-merge. It
does not auto-push. It does not auto-deploy. The human is always in the
loop.

**Law 4: the project documents itself for its own agents.** The skill at
`.agents/skills/chasseur-onirique/SKILL.md` is a markdown file that any
agent will load before touching the codebase. It tells them the four
laws, the header convention, the canonical commands, the safety rules,
the file map, the known pitfalls. A future agent that violates one of
those laws has ignored a file it was told to read. That's a real failure
mode, and the meta-skill is the defence against it.

These four laws are not in the README because they're the README. They're
in the meta-skill because they need to be inside the system, not outside
it.

## The deny-list, and what it costs

`basher` is the only agent that can mutate the world. It runs shell
commands. It's gated by three things, in order:

1. **Dry-run by default.** `process.env.CHASSEUR_ONIRIQUE_LIVE === '1'`
   is the only way to actually execute. If the env var is unset — and
   the test suite makes sure it stays unset in CI — every `basher` call
   returns a dry-run transcript.
2. **A deny-list of 7 regexes.** Three for the `rm` family (`rm -rf /`,
   `rm ... -rf /`, `rm ... --no-preserve-root /`), then `curl|sh`,
   `wget|sh`, `sudo`, `dd if=`. Each regex is commented with its
   rationale and the attacks it catches.
3. **A documented gap.** `rm -r -f /` — split flags, no contiguous `-rf`
   substring — is **not** caught. By design. The deny-list file
   (`docs/deny-list.md`) calls this out explicitly. The comment in the
   source says "KNOWN GAP" in capital letters.

The trade-off here is the one that every safety system eventually has to
make: a deny-list can either be tight and full of false positives, or
loose and full of false negatives. I chose loose. The reason is that
`basher` is a *seam*, not a feature — the day a real LLM plugs into
`PlannerStrategy`, the LLM will propose shell commands, and the deny-list
is the last line of defence. A deny-list that fires on `rm -r` (no flag)
will block legitimate cleanup commands. A deny-list that fires on
`rm -r -f /` is precise and lets everything else through, at the cost of
this one gap.

The gap is real. I know it's real. The file is there so nobody can claim
they weren't told.

## Why HeuristicPlanner and not an LLM

The honest answer: because I wanted to ship something I understood. The
LLM version of `PlannerStrategy` would take a prompt, hit the Anthropic
API, and return a plan. That's a 30-line file. I could write it in 20
minutes. I didn't, because:

- I can't test a non-deterministic planner. The 65 tests in the project
  are deterministic. A planner that flips a coin on the same prompt
  twice would break the `vitest` snapshots. The seam is there for
  someone else to wire in, but I want the system to be *reliable* in the
  state I ship.
- The heuristic planner is **good enough for the demo.** The four-step
  plan (gather context → think → act → review) covers 80% of what a
  user would ask. The 20% it misses is exactly the kind of thing a
  human will adjust the prompt for.
- The cost of getting it wrong is high. A planner that decides
  "investigate the bug" when the user said "refactor the function" is
  annoying. A planner that decides "delete the cache directory" is
  catastrophic. The deny-list guards the second; the heuristic
  planner avoids the first.

The seam is real. A future commit can drop in `LLMPlanner` and the
system will run. But the seam is not a promise — it's an invitation.

## The meta-skill, and the recursion

The meta-skill is the most interesting file in the project. It's a
SKILL.md that documents the project's own invariants. The orchestrator
loads it like any other skill, surfaces it in the "Loaded skills:" log,
and — if a real LLM ever plugs in — the body becomes part of the
system prompt.

The recursion is the point. A project that documents itself for its own
agents is harder to drift. If the orchestrator decides to add a new
agent and forgets to update the registry, the meta-skill catches it. If
the deny-list changes, the meta-skill catches it. If a new file is
added without the 4-line credit header, the meta-skill catches it. The
docs are inside the system, so the system has to honour them to remain
consistent with itself.

The limit is the same as the law: the meta-skill is only as good as the
agents that load it. If a future agent skips the skill manifest — if
the orchestrator ever bypasses `loadSkills` — the recursion breaks. But
the test `src/skills.test.ts` pins the manifest shape, so the bypass
would have to be intentional.

## The debt I'm taking on

- **No LLM in the loop.** The planner is heuristic. The synthesizer is
  structural. Every agent is pure. Replacing any of these with an LLM
  call is a future commit. The seam is there. The debt is the test
  coverage of the seam.
- **No persistence across runs.** Context is a 50-message window. When
  the session ends, the context is gone. The `snapshot-session.ts`
  script is the workaround: capture a session, write it to disk, learn
  from it. But the orchestrator itself does not remember anything
  between invocations.
- **No concurrency control on the deny-list.** If a future LLM planner
  proposes `rm -r -f /` and the orchestrator runs it, the deny-list
  catches it (well, except the gap). If the orchestrator runs the
  commands in parallel, the order of deny-list checks is still
  per-command, so there's no race. But there's no defence against a
  future maintainer who adds a "deny-list bypass for trusted prompts"
  flag. The test suite doesn't guard against that, and the meta-skill
  says nothing about it. **This is the biggest gap in the project.**
- **No real benchmarks.** The "8 files, 65 tests, all green" is
  coverage, not quality. The tests pin the shape, not the usefulness.
  The snapshot tool makes it cheap to add regression tests, but I
  haven't written a single "did the agent actually find the bug"
  test. That's the next thing I'd add.

## What's next

The next thing I'd add is a `self-check` command that runs the
orchestrator on the project itself, with a prompt asking it to find
places where the docs diverge from the code. The report goes to
`docs/audits/<ISO>.md`. Exit 0 if no divergence, exit 1 otherwise. It's
the dogfooding version of the meta-skill: a system that audits itself.

That’s the next devlog.

## Postscript: the `self-check` was added (v0.2.1), and it took a debugging session to ship

The script landed in the repo a few hours after this devlog. It worked
on the first try… or so I thought. The first run exited 1 with no
output. Typecheck passed, the orchestrator worked in the snapshot CLI
with the same imports, and a fresh minimal file at the same path
worked fine. The `process.exit grep` showed no hidden exit in any
agent. The mystery held for a full debugging session.

The cause was a JSDoc comment that contained `src/**/*.ts`. esbuild
parses `**/` (two stars then a slash) as a JSDoc comment terminator,
because `*/` ends a comment and `**` + `/` contains that pattern. The
JSDoc ended prematurely at `src/**`, the rest of the line was parsed
as code, and the script crashed during transpilation. The catch block
in `main()` never ran because the crash happened before the module
loaded.

The fix was one line: rephrase the JSDoc to avoid `**/`. The lesson
isn't really about JSDoc — it's about silent failures. A TypeScript
file can be syntactically broken in a way that `tsc` accepts (the
TypeScript compiler is more forgiving than esbuild) but esbuild
rejects. And esbuild, when run from a path it’s already seen, can
swallow the error and return exit 1 with no output. The whole project
is a reminder that **a passing typecheck is not a passing build**, and
that every script should be exercised end-to-end before it ships.

The devlog is the meta-skill’s mirror. The meta-skill said “before any
action, commit to a plan.” The devlog said “the next thing I'd add is
self-check.” The actual debugging required neither plan nor foresight —
just patience and `process.stderr.write` at every line. Sometimes
that’s all engineering is.
