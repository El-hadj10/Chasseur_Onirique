### Phase 2 — history rewrite

Two early commits carried `Co-Authored-By: Claude <noreply@anthropic.com>` trailers at the bottom of their messages. Those trailers were the cause of "Claude shows up in the Contributors panel".

I ran `git filter-branch --msg-filter` on a working branch (`scrub/claude-trailers`) with a perl one-liner that removed the offending line and re-collapsed double blank lines. Trees stayed byte-identical; message bodies and SHAs changed. The 14 commits reachable from the working branch were rewritten.

Then a `git push --force origin scrub/claude-trailers:main` made the scrubbed history the new `main`. After that, the GitHub REST `/contributors` endpoint correctly returned `["El-hadj10"]`.

### Phase 3 — Claude *still* showed in the Contributors panel

Useful, expected. But the **Insights → Contributors** UI kept showing Claude. Cause:

> `refs/tags/v0.2.0` on origin still pointed at `e60dfb9` — the original pre-rewrite commit. GitHub UI aggregates trailer co-authors from *all* reachable refs, not just the default branch.

`filter-branch` doesn't touch tags. Force-push of a head ref doesn't either. The tag was a **zombie**: untouched for years, still pointing at a commit that had every line of the trailer we just spent Phase 2 removing.

Three options were considered:

- **(a)** Move tag `v0.2.0` to a rewritten SHA and force-update it.
- **(b)** Delete `v0.2.0` and (separately) re-create it on a clean commit.
- **(c)** **Nuke the GitHub repo + recreate from the local clone.**

We picked **(c)**. The reasoning: nothing cleans Insights, Releases, the wiki (if any) and the contributor cache harder than a fresh repo with the same name. The four old version tags (`v0.2.0`, `v0.2.1`, `v0.3.0`, `v0.3.1`) were lost in the recreate — that loss was acceptable because they were tied to the deleted repo's metadata and re-tagging is a clean follow-up.

### Phase 4 — local hygiene before the recreate

Before recreating, the **local clone** itself had to be cleaned. Otherwise the first push after recreate would re-introduce the same ghosts via stale local refs:

- Reset `refs/heads/main` onto the clean scrubbed tip (`625e88f`).
- Deleted local branch `backup/origin-main-pre-scrub` (a safety-net we'd created to recover the diverged commits).
- Deleted local `refs/tags/v0.2.0` (which had been **resurrected** by an earlier `git fetch --prune` step — fetch was pulling in the dirty tag again from origin).
- Crucially: did **not** run `--tags` in the subsequent push. Only `git push -u origin main`. Pushing tags by accident would have re-shipped the trailer-bearing object.

### Phase 5 — recreate + push

- Deleted the GitHub repo via the UI (the OAuth token in `gh` did not have the `delete_repo` scope; the CLI option was therefore unavailable and we did the delete in the browser).
- `gh repo create El-hadj10/Chasseur_Onirique --public --description "..."` on the freshly-free name (GitHub took ~30 s to propagate; we waited and re-checked via `gh repo view`).
- `git push -u origin main`. 12 commits. All trailer-clean.
- API post-check: `/contributors` = `[{login: "El-hadj10", contributions: 12}]`. `/tags` = `[]`. 0 commit bodies contained `Claude` or `noreply@anthropic.com`.

## What I deliberately did NOT do

- **Re-tag `v0.2.x` / `v0.3.x` on the new repo.** Those tags were tied to the deleted repo and its commit SHAs. Re-tagging is a follow-up, not a recovery.
- **Preserve old commit SHAs in the new repo.** `filter-branch` rewrites trailers but the SHA inevitably changes (because the message bytes changed). Trying to keep the old SHAs would have meant keeping the trailers.
- **Run `git gc --prune=now --aggressive` immediately.** The local clone still has dangling objects from `filter-branch`. A `git gc` cleanup is a separate maintenance action; not required for the push.

## Lessons

The most durable lessons of this rebuild:

1. **GitHub UI contributors ≠ GitHub REST contributors.** They are two different views of overlapping data, with different cache lifetimes. Trust the REST API for verified state; trust the UI slowly after force-pushes.
2. **Tags are the silent culprit in contributor leakage.** A branch rewrite does nothing to a tag pointing at an untouched commit. Always audit `refs/tags/*` on origin, not just `refs/heads/*`.
3. **Nuke + recreate is the cleanest exit.** When *any* of (a) tags, (b) Releases, (c) Insights, or (d) Discussions cache contains stale references, recreating the repo under the same name is cheaper than untangling each source.
4. **`filter-branch` > `filter-repo` is mostly historical.** `git filter-repo` is faster and produces fewer backup refs, but requires install. `filter-branch` is built-in and good enough for this scale.
5. **The author's manual is the safety story.** This devlog + a future `CONTRIBUTOR_HYGIENE.md` are what prevent the next contributor (human or AI) from quietly re-adding a `Co-Authored-By:` line that breaks everything.
